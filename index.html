Regulatory

IEC 62304 medical device software lifecycle

Not documentation as a checkbox — the full software lifecycle process, applied to real kernel drivers and a production ROS 2 package. Public and auditable.

SDP
Software Development Plan
Development process, tools, configuration management, and verification strategy per IEC 62304 §5.1.
SCMP
Software Configuration Management Plan
Version control strategy, branch policy, release tagging, and change control per §5.1.9.
SPRP
Software Problem Resolution Plan
Defect classification, severity assessment, and corrective action process per §9.
FMEA
Failure Mode & Effects Analysis
ISO 14971 software-level risk analysis: failure modes, severity, probability, risk controls per §7.1.2.
TM
Traceability Matrix
Bidirectional traceability: SRS ↔ software design ↔ unit tests ↔ integration tests per §5.5.
SOUP
SOUP Records
IEC 62304 §8.1.2 records for each Linux kernel driver: title, author, version, Kconfig dependencies, known anomalies.

The kernel community has already built the scaffolding for §8.1.2 a) and c). Every upstream driver declares MODULE_AUTHOR, MODULE_DESCRIPTION, MODULE_LICENSE, and a Kconfig dependency graph. The git history provides a reproducible version identifier. get_maintainer.pl gives the CVE contact chain. What's missing is the tooling that bridges kernel developer conventions to regulatory terminology — and closing that gap makes Linux a more defensible choice for Class B and Class C medical device software.
